先提个问题：为什么大模型不能依据你自己公司的私有组件库生成代码？

答：由于大模型的*训练数据集*不包含你公司的私有组件数据，因此不能够生成符合公司私有组件库的代码。

因此，解决问题的核心就是：让大模型知道你公司的私有组件库是什么样的。

要想让大模型知道你公司的私有组件什么样的有三种方式：
1. 预训练：从 0 ～ 1 ，预训练一个属于你自己的基座模型（自定义训练数据集），效果相对最好，模型能完全理解私有组件，但是
  + 成本极高（算力、时间、人力）
  + 技术门槛高
  + 需要海量训练数据
  + 维护成本高
2. 微调：成本相对较低，只需少量标注数据（相对预训练来说），可以快速适应特定场景，但是
   + 效果不如预训练
   + 可能出现灾难性遗忘
   + 仍需要一定的算力和专业知识
3. RAG，优点：
   + 实现简单，成本最低
   + 无需训练，可即时更新知识
   + 可控性强，易于维护
   可以保证知识的准确性
  <br />
  缺点
   + 受限于上下文窗口大小
   + 检索质量依赖于向量化效果
   + 响应速度可能较慢
**RAG**
  图示构建 RAG 向量知识库的过程
  <img width="1329" height="490" alt="image" src="https://github.com/user-attachments/assets/a8b37829-ba20-40ff-9e94-27bd5812c767" />
1. 原始数据（Resource Data）:从各种来源收集原始数据，比如公司私有组件库的文档文本。
2. 分块（Chunking）:将资源数据细分为更小的块，称为Chunk。
3. 向量化（Embedding）:将每个Chunk转换为向量表示，便于后续根据向量进行语义相似度匹配。
4. 存储至向量数据库:将所有的Chunk和Embedding一一对应存储在向量数据库中，用于后续向量匹配检索出原始的 Chunk 数据。


**RAG 向量检索过程的简单示例**
<img width="7044" height="964" alt="image" src="https://github.com/user-attachments/assets/3a1cd3f6-33e9-45d1-998e-8871e29ca572" />
1. 用户输入一个问题，如：帮我生成一个table，包含姓名、年龄、性别。
2. 将问题转换为向量表示。
3. 将用户需求的向量和向量数据库中的向量进行相似度匹配，检索出相似度高的数据源（Retrieval）。
4. 将检索出的数据源和用户需求的问题组合（Augmented），一起输入给大模型（Generation）。

评估器模块（Evaluator）：
1. 对检索结果进行质量评估
2. 将知识分类为：相关知识、存疑知识、不相关知识
3. 通过评估确保输入到大模型的知识质量
4. 减少"幻觉"的产生，提高答案准确性
5. 作为知识分流的决策点，确定后续处理路径

问题重写模块（Query Rewriter）：

1. 针对存疑/不相关知识进行问题重写
2. 优化查询表达，提高检索准确度
3. 通过重写扩展问题维度，获取更多相关上下文
4. 实现查询的自我纠正和优化
5. 作为知识补充的入口，提供二次检索机会

重检索模块（Re-Searcher）：

1. 执行重写后问题的二次检索
2. 补充和扩展知识库的查询范围
3. 提供多轮检索的可能性
4. 与原始检索形成互补，提高召回率
5. 作为知识获取的最后一环，确保答案的完整性

这种模块化设计的优势：

1. 各模块职责明确，可独立优化和替换
2. 形成完整的知识评估-重写-检索闭环
3. 通过多重保障机制提高最终答案质量
4. 灵活应对不同质量的检索结果
5. 支持系统的持续优化和迭代
